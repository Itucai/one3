简介

在大型语言模型的输入输出机制中，文本提示（有时会配合图像等多模态输入）是模型用于生成预测结果的基础输入。您无需成为数据科学家或机器学习工程师，任何人都可以编写提示词。但要想构建真正高效的提示词，这个过程可能相当复杂。提示词的效果受多重因素影响，所选模型版本、模型的训练数据、参数配置、措辞精准度、语言风格与语气、结构设计以及上下文关联性都至关重要。因此，提示词工程本质上是一个需要持续优化的过程。低质量的提示词不仅会导致模棱两可或错误的输出，更会限制模型生成有价值内容的能力。

当你与Gemini聊天机器人交流时，你实际上是在编写提示。不过本白皮书将重点探讨如何在Vertex AI平台或通过API为Gemini模型编写提示词，因为直接调用模型时，您将能访问温度参数（temperature）等完整配置项。

这份白皮书详细讨论了提示工程。我们将探讨各种提示技术，帮助你入门并分享成为提示专家的技巧和最佳实践。我们还将讨论设计提示时可能遇到的挑战。

提示工程

记住LLM的工作原理，它是一个预测引擎。模型接收序列文本作为输入，然后根据其训练数据预测下一个标记。LLM会反复执行此操作，将先前预测的标记添加到序列文本末尾以预测下一个标记。下一个标记的预测基于先前标记的内容与LLM在训练期间所见内容之间的关系。

编写提示时，你是在为LLM设置正确的标记序列。提示工程是设计高质量提示的过程，引导LLM生成准确的输出。这包括调整提示以找到最佳版本，优化提示长度，并评估提示的写作风格和结构是否与任务相关。在自然语言处理和LLM的背景下，提示是为模型提供的输入，用于生成响应或预测。

这些提示可用于完成各种理解和生成任务，如文本摘要、信息提取、问答、文本分类、语言或代码翻译、代码生成以及代码文档或推理。

请随时参考Google的提示指南，其中包含简单有效的提示示例。

进行提示工程时，首先需要选择一个模型。无论你使用Vertex AI中的Gemini语言模型、GPT、Claude，还是像Gemma或LLaMA这样的开源模型，提示可能需要针对你的特定模型进行优化。

除了提示外，你还需要调整LLM的各种配置。

LLM输出配置

选择模型后，你需要确定模型的配置。大多数LLM提供多种配置选项来控制输出。有效的提示工程需要根据任务优化这些设置。

输出长度

一个重要的配置参数是控制生成响应中的token数量。生成更多token需要大语言模型进行更多计算，从而导致能耗增加、响应时间可能变慢以及成本上升。

减少LLM的输出长度并不会使LLM在输出内容上变得更加简洁或风格化，它只是让LLM在达到长度限制时停止继续预测token。如果您的需求要求输出较短，您可能还需要对提示词进行专门设计来满足要求。

输出长度限制对于某些LLM提示技术尤为重要，比如ReAct，LLM可能会在您想要的响应之后继续生成无用的token。

采样控制

LLM不会预测单个标记，而是预测下一个标记的概率分布。温度、Top-K和Top-P是最常见的配置选项，用于决定如何从预测的标记概率中选择输出标记。

  • 温度

温度参数控制token选择的随机程度。较低温度适合需要确定性输出的提示，而较高温度可能产生更多样化或出人意料的结果。温度为0（贪婪解码）时是完全确定性的：总是选择概率最高的token（但需注意，若两个token具有相同的最高预测概率，根据平局处理机制的不同，温度设为0时输出也可能不一致）

温度接近最大值时，输出往往更加随机。随着温度不断升高，所有token被选为下一个预测token的概率会趋于均等。

Gemini温度控制机制与机器学习中使用的softmax函数原理相似，低温设置类似于softmax低温（T），强调单一优选结果，确定性高。高温设置类似于softmax高温，接受更宽泛的结果范围 这种灵活性特别适用于不需要严格精确输出的场景（例如创意内容生成实验）。

  • Top-K与Top-P采样

Top-K和Top-P（又称核采样）是LLMs中用于限制预测token范围的两种采样设置，与温度参数类似，它们控制生成文本的随机性和多样性。

  • Top-K采样从模型预测分布中选择概率最高的K个token。K值越大，输出越具创造性，K值越小，输出越保守准确，K=1时等同于贪婪解码。
  • Top-P采样选择累计概率不超过设定值（P）的最高概率token集合。P值范围，0（贪婪解码）到1（包含所有vocabulary token）。

最佳实践是通过实验比较两种方法（或组合使用），根据实际需求选择最优方案。

整合一切

在top-K、top-P、temperature和要生成的token数量之间进行选择取决于具体应用和预期结果，这些设置都会相互影响。同时，确保您理解所选模型如何将不同的采样设置组合在一起也很重要。

如果temperature、top-K和top-P都可用（如在Vertex Studio中），同时满足top-K和top-P标准的token将成为下一个预测token的候选，然后应用temperature从这些通过top-K和top-P标准的token中进行采样。如果只有top-K或top-P可用，行为相同但仅使用其中一个top-K或top-P设置。

如果temperature不可用，则从满足top-K和/或top-P标准的token中随机选择以生成单个下一个预测token。

当某个采样配置值设为极端值时，该采样设置要么会抵消其他配置设置，要么变得无关紧要。

  • 如果你将温度设置为 0，top‑K 和 top‑P 就变得无关紧要了，最可能的标记将成为下一个预测的标记。如果你将温度设置得非常高（超过 1— 通常在 10 以上），温度就变得无关紧要了，然后随机采样通过 top‑K 和 / 或 top‑P 标准的标记来选择下一个预测的标记。
  • 如果你将 top‑K 设置为 1，温度和 top‑P 就变得无关紧要了。只有一个标记通过了 top‑K 标准，并且这个标记就是下一个预测的标记。如果你将 top‑K 设置得非常高，比如达到 LLM 词汇表的大小，任何有非零概率成为下一个标记的标记都将满足 top‑K 标准，并且没有任何标记被选中。
  • 如果你将 top‑P 设置为 0 （或一个非常小的值），大多数 LLM 采样实现将只考虑最可能的标记来满足 top‑P 标准，这使得温度和 top‑K 变得无关紧要。如果你将 top‑P 设置为 1，任何有非零概率成为下一个标记的标记都将满足 top‑P 标准，并且没有任何标记被选中。

作为通用起点，温度为0.2、top-P值为0.95、top-K值为30的配置能产生相对连贯且适度富有创意的结果。若需要特别具有创造性的输出，可尝试温度0.9、top-P 0.99配合top-K 40的初始设置。而对于创意要求较低的场景（如追求稳定性），建议采用温度0.1、top-P 0.9和top-K 20的组合。需注意，当任务存在唯一正确答案时（例如数学问题求解），应将温度参数设置为0以获得确定性输出。

注意：随着自由度的增加（更高的温度、 top‑K 、 top‑P 和输出令牌），LLM 可能会生成与主题关联度较低的文字。

警告：您是否见过以大量填充词结尾的响应？这也被称为”重复循环错误”，这是大语言模型中的一个常见问题。当模型陷入循环时，会反复生成相同的（填充）单词、短语或句子结构，这种情况通常因不恰当的temperature和top-k/top-p设置而加剧。这种现象可能出现在低temperature和高temperature两种设置下，但原因不同，在低temperature时，模型变得过于确定，严格遵循最高概率路径，如果该路径重新访问之前生成的文本，就可能导致循环。在高temperature时，模型的输出变得过于随机，由于可选选项众多，随机选择的单词或短语可能偶然回到之前的状态，从而形成循环。在这两种情况下，模型的采样过程都会”卡住”，产生单调无用的输出，直到填满输出窗口。解决这个问题通常需要仔细调整temperature和top-k/top-p值，以在确定性和随机性之间找到最佳平衡。

提示技术

大语言模型（LLMs）经过专门调优以遵循指令，并通过海量数据训练使其能够理解提示词并生成回答。但LLMs并非完美无缺。您的提示文本越清晰，模型就越能准确预测后续文本。此外，充分利用LLMs的训练机制和运作原理的特定技巧，将帮助您从模型中获得更精准的相关结果。

现在我们已经了解了什么是提示工程以及它需要什么，让我们深入探讨一些重要的提示技术示例。

通用提示 / 零样本

零样本提示是最简单的提示类型。它仅提供任务描述和供LLM开始处理的基础文本。这种输入可以是任何内容：一个问题、故事开头或指令。”零样本”这一名称代表”无示例”的含义。

当零样本（zero-shot）方法无效时，您可以在提示词中提供示范案例，从而采用”单样本（one-shot）”或”少样本（few-shot）”提示技术。

单样本 & 少样本

在为 AI 模型创建提示时，提供示例是有帮助的。这些示例可以帮助模型理解你要求的内容。当你想引导模型到一个特定的输出结构或模式时，示例特别有用。

单样本提示提供了一个  示例，因此得名单样本。其理念是模型有一个可以模仿以最好完成任务的示例。

少样本提示提供了多个示例给模型。这种方法向模型展示了它需要遵循的模式。理念与单样本类似，但提供多个目标模式的示例增加了模型遵循该模式的可能性。

少样本提示（few-shot prompting）所需示例数量取决于几个因素，包括任务复杂性、示例质量以及所使用的生成式AI（gen AI）模型的能力。一般来说，您应该至少使用三个到五个示例进行少量样本提示。然而，对于更复杂的任务，您可能需要使用更多的示例，或者由于模型输入长度限制，您可能需要使用更少的示例。

系统、上下文和角色提示

系统、上下文和角色提示都是用于指导 LLM 生成文本的技术，但它们关注不同的方面：

  • 系统提示（System prompting）为语言模型设定整体上下文和目的。它定义了模型应该执行的”大局”任务，如翻译语言、分类评论等。
  • 上下文提示（Contextual prompting）提供与当前对话或任务相关的具体细节或背景信息。它帮助模型理解所提要求的细微差别，并相应调整回答。
  • 角色提示（Role prompting）为语言模型指定特定的角色或身份。这有助于模型生成与所分配角色及其相关知识和行为一致的响应。

系统、上下文和角色提示之间可能存在相当大的重叠。例如，一个为系统分配角色的提示也可以包含上下文。每种提示类型的主要目的略有不同：

  • 系统提示：定义模型的基本能力和总体目标。
  • 上下文提示：提供即时的、特定任务的信息来指导响应。它是高度特定于当前任务或输入的，具有动态性。
  • 角色提示：塑造模型的输出风格和语气。它增加了特定性和个性层面。

区分系统提示、上下文提示和角色提示为设计具有明确意图的提示词提供了一个框架，允许灵活组合，并更容易分析每种提示类型如何影响语言模型的输出。

回溯提示

回溯提示是一种通过提示大语言模型（LLM）先思考与当前具体任务相关的普遍性问题，然后将该普遍性问题的答案输入到后续具体任务提示中，从而提高性能的技术。这种回溯步骤能让LLM在尝试解决具体问题前，激活相关的背景知识和推理过程。

通过考虑更广泛和根本的原则，LLM能生成更准确和富有洞察力的响应。回溯提示鼓励LLM进行批判性思考，并以新颖创造性的方式应用其知识。相比直接提示LLM执行任务，该技术通过利用LLM参数中更多未被调用的知识，改变了最终执行任务的提示方式。

回溯提示通过聚焦普遍原则而非具体细节，有助于减轻LLM响应中的偏见。

思维链（ CoT ）

思维链（Chain of Thought，简称CoT）是一种通过生成中间推理步骤来提高大型语言模型（LLM）推理能力的技术。这有助于LLM生成更准确的答案。你可以将其与少量示例提示（few-shot prompting）结合使用，以在需要推理才能响应的更复杂任务上获得更好的结果，因为这是零次示例（zero-shot）思维链的挑战。

CoT有很多优点。首先，它在低努力的情况下非常有效，并且与现成的LLM（因此无需微调）配合得很好。你还可以通过CoT提示获得可解释性，因为你可以从LLM的响应中学习并看到所遵循的推理步骤。如果出现故障，你将能够识别它。思维链似乎在不同LLM版本之间移动时提高了鲁棒性。这意味着你的提示在不同LLM之间的性能变化应该比不使用推理链的提示要小。当然，也有一些缺点，但它们是相当直观的。

LLM的响应包括思维链推理，这意味着更多的输出标记，这意味着预测成本更高，耗时更长。

自洽性

大语言模型（LLM）虽然在各种NLP任务中取得了显著成功，但其推理能力仍被视为一个无法仅通过扩大模型规模来克服的局限。正如我们在前文思维链提示（Chain of Thought prompting）部分所学，可以通过提示让模型像人类解决问题一样生成推理步骤。然而，CoT使用的是简单的”贪婪解码”（greedy decoding）策略，限制了其效果。自洽性方法（Self-consistency）结合了采样和多数表决机制，通过生成多样化的推理路径并选择最一致的答案，提高了LLM生成响应的准确性和连贯性。

自洽性方法通过以下步骤实现：

  • 生成多样化推理路径：向LLM多次提供相同的提示，通过设置较高的温度参数促使模型针对问题生成不同的推理路径和视角。
  • 从每个生成的响应中提取答案。
  • 选择出现频率最高的答案。

让我们看一个邮件分类系统的例子，该系统需要将邮件分类为”重要”或”不重要”。我们会多次向LLM发送零样本思维链提示，观察每次提交后的响应是否不同。注意邮件中使用的友好语气、措辞选择和讽刺表达，这些都可能会误导LLM。

您可以使用上述提示词测试其是否返回一致的分类结果。根据所使用的模型及温度参数配置，系统可能返回”IMPORTANT”或”NOT IMPORTANT”两种判定。

通过生成多个思维链（Chains of Thoughts）并选择最常出现的答案（”IMPORTANT”），我们可以从大语言模型（LLM）获得更一致且正确的答案。

这个例子展示了自洽性提示（self-consistency prompting）如何通过考虑多个视角并选择最一致的答案，来提高LLM响应的准确性。

这种方法使 ToT 特别适合需要探索的复杂任务。它通过维护一个思维树来实现，其中每个思维代表一个连贯的语言序列，作为解决问题的一个中间步骤。然后，模型可以通过从树的不同节点分支出来探索不同的推理路径。

ReAct （推理与行动）

推理与行动（ReAct）提示是一种范式，使大语言模型（LLM）能够通过结合自然语言推理和外部工具（搜索、代码解释器等）来解决复杂任务，允许LLM执行某些操作，例如与外部API交互以检索信息，这是实现智能体建模的第一步。

ReAct模仿了人类在现实世界中的操作方式，即我们通过语言进行推理，并可以采取行动获取信息。在各种领域中，ReAct的表现优于其他提示工程方法。

ReAct提示的工作原理是将推理和行动结合成一个”思考-行动”循环。LLM首先对问题进行推理并生成行动计划，然后执行计划中的行动并观察结果。LLM随后利用这些观察结果更新其推理并生成新的行动计划。这个过程持续进行，直到LLM找到问题的解决方案。

一个好的经验法则是从 6 个少量示例开始，并测试准确性。

必须及时掌握模型架构变更、新增数据和功能更新。尝试使用更新的模型版本，并调整提示词以更好地利用新模型特性。像Vertex AI Studio这样的工具非常适合存储、测试和记录不同版本的提示词。

尝试不同的输出格式

除了提示词的输入格式外，还应尝试不同的输出格式。对于数据提取、选择、解析、排序、排名或分类等非创造性任务，可以尝试让输出以结构化格式（如JSON或XML）返回。

从提示词中返回JSON对象有以下优势：在实际应用中，无需手动创建JSON格式，数据可以直接按排序顺序返回（在处理日期时间对象时特别方便）。最重要的是，通过要求JSON格式，可以强制模型生成结构化输出，减少幻觉内容的产生。

总结来说，使用 JSON 作为输出格式的好处：

  • 始终以统一格式返回数据
  •  精准聚焦目标数据内容
  • 降低幻觉产生概率 
  • 保持数据关联性 
  • 自动获取数据类型 
  • 支持排序功能

JSON 修复

虽然以JSON格式返回数据具有诸多优势，但也存在一些缺点。JSON的结构化特性虽然有利于解析和应用集成，但相比纯文本需要消耗更多的token，这会导致处理时间增加和成本上升。此外，JSON的冗长特性很容易占满整个输出窗口，当生成内容因token限制被突然截断时，这一问题尤为突出。这种截断通常会导致JSON格式无效——缺失关键的闭合括号或中括号，从而使输出无法使用。幸运的是，像json-repair这样的工具库（可通过PyPI获取）在这种情况下非常有用。该库能智能地尝试自动修复不完整或格式错误的JSON对象，使其成为处理大语言模型生成的JSON数据时的重要工具，特别是在应对可能出现的截断问题时尤其有效。

使用模式

正如本文多次提到的，使用结构化JSON作为输出是一个很好的解决方案。但输入呢？虽然JSON非常适合构建LLM生成的输出结构，但它对于构建您提供的输入数据也同样非常有用。这就是JSON Schema发挥作用的地方。JSON Schema定义了您输入的JSON数据的预期结构和数据类型。通过提供一个schema，您为LLM提供了一个清晰的数据蓝图，帮助它聚焦于相关信息，并降低误解输入的风险。此外，schema还能帮助建立不同数据片段之间的关系，甚至通过包含特定格式的日期或时间戳字段，使LLM具备”时间感知”能力。

一个简单的例子：让我们假设您想使用一个大型语言模型（ LLM ）来生成电子商务目录中产品的描述。除了提供产品的文本描述外，您还可以使用 JSON 模式来定义产品的属性：

{
  "type": "object",
  "properties": {
    "name": { "type": "string", "description": "Product name" },
    "category": { "type": "string", "description": "Product category" },
    "price": { "type": "number", "format": "float", "description": "Product price" },
    "features": {
      "type": "array",
      "items": { "type": "string" },
      "description": "Key features of the product"
    },
    "release_date": { "type": "string", "format": "date", "description": "Date the product was released" }
  }
}

CoT 最佳实践

必须将答案放在推理过程之后，因为推理过程的生成会改变模型预测最终答案时接收到的token。

使用CoT和自我一致性方法时，需要能够从提示中提取最终答案，并将其与推理过程分离。

对于CoT提示，温度参数应设置为0。

思维链提示基于贪婪解码策略，即根据语言模型分配的最高概率预测序列中的下一个词。一般来说，当使用推理得出最终答案时，很可能只有一个正确答案。因此温度参数应始终设为0。

详细记录提示尝试

虽然本章前文已提及，但我们必须再次强调其重要性，请完整详细地记录您的提示尝试，这样才能持续了解哪些方法有效、哪些无效。

提示输出可能在不同模型、不同采样设置以及同一模型的不同版本之间有所不同。此外，即使在同一模型的相同提示下，输出句子结构和词语选择也可能存在细微差异。（例如，如前所述，如果两个标记具有相同的预测概率，则可能会随机打破平局。这可能会影响后续预测的标记。）
